# コーディング規則

## 基本

- cmoc は python で実装する
- python コーディングスタイルは PEP 8 を遵守する
- 命名、責務、入出力を明確に保つ
- コードの意味的な小さいブロックごとに、そのブロックで何をしているかのコメントを記入する
- コード単体では意図が伝わりにくい場合は `NOTE` を付けてコメントを書く
- オーバーエンジニアリングは避け、要求を満たす最小限の変更に留める

## 型ヒント

- 型ヒントは必ず書く
- 必要以上に `Any` を使わない
- `from __future__ import annotations` は使わない
    - 型チェッカーと実行時挙動を揃えるためのルールである

## import のルール

- `<cmoc-root>/src/` をルートとした相対 import が可能であることを前提とする
- 可能なら `.` を起点とする相対 import を行う
- 循環参照は、まずモジュール分割や構造調整で回避を試みる
- どうしても回避不能な場合のみ `TYPE_CHECKING` を使ってよい

## docstring

- 関数・クラスなどには docstring を書く
- docstring は google style に従う
- シグネチャを見れば分かる情報を冗長に繰り返して、無駄に縦長にしない
- コードだけでは読み取りにくい「その関数・クラスが何者か」を優先して説明する

## コメントと言語

- コメントは基本的に日本語で書く
- 変数名・型名などの識別子は日本語に訳さなくてよい
- 英単語のままの方が通じやすいなら、無理に日本語化しない
- ログメッセージは英語で書く
    - 文字コード由来のトラブルシューティング上の問題を避けるため
- 関数内の適切なコードの塊ごとに、その塊で何をしているのかの簡潔な説明をコメントとして書く
    - 関数の先頭コードブロックについては、 `"""` による docstring と混同して `#` によるコメントを付け忘れたりしないように注意すること
- 実装だけからでは理解するのが難しい補足は NOTE でマークしてコメントとして書く

## 非公開の識別子

- 非公開とは
    - モジュール外に公開しない
    - クラス外に公開しない
    - ...
- そういった非公開の識別子は先頭に `_` を付けることする
