# realization standards

## 前提

- realization files の基本情報は `<cmoc-root>/oracles/docs/app_specs/realization/basics.md` を参照すること
- realization standards とは realization files の内容が標準的に従うべき事項である
- cmoc による realization files 編集作業 (e.g. `cmoc apply` サブコマンド) の実装は realization standards と整合しなければならない
- `<cmoc-root>` 配下の realization files それ自身もまた realization standards に従わなければならない

## standards

### realization files の総文字数の最小化を目標とする

- 前提
    - 実装担当者である AI に与える課題の難易度という観点で realization files の規模は小さい方が望ましい
    - トークン消費という観点で realization files の規模は小さい方が望ましい
- 規則
    - 必ず守らなければならない要件を満たしている範囲内（解空間内）で、realization files 全体で見た時の総文字数が最小となること（文字数最小解）を目指す
- e.g.
    - 複数個所で出現するよく似た処理は関数化し、それを各所で使い回すべきである
    - 意味的に重複している実装は１つに集約するべきである
    - 現行の仕様と関係のない、過去の仕様に基づく実装は削除されるべきである

### realization files の高品質化は望ましいことである

- 前提
    - realization files は cmoc により継続的にメンテナンス (i.e. `cmoc apply` サブコマンドによって oracles に反映) される
    - メンテナンスにかかるコスト (i.e. AI が消費するトークン数) の観点から realization files の品質は高いほうが望ましい
    - realization files の高品質化は、単なる行数削減ではなく、変更容易性・責務の明確さ・不要部分の少なさを含む
- 規則
    - realization files は、現行仕様を満たすために必要な情報だけを持つべきである
    - realization code は、責務・入力・出力・副作用が読み取りやすい単位に分割されるべきである
    - ただし、読み取りやすさを損なう過度な圧縮や、将来使うかもしれないだけの抽象化は禁止する
    - コメント・docstring・補助ファイルは、実装から読み取れない意図または制約を説明するためにだけ使う
    - 現行仕様・現行実装から不要になった realization files、関数、クラス、分岐、設定、テスト、コメントは削除する
- e.g.
    - 複数個所で出現するよく似た処理を関数化し、それを各所で使い回すべきである
    - コード・実装の意味的なブロックごとに、なぜその実装でなければいけないのか？　その根拠をコメントとして書くべきである
    - 仕様変更により使われなくなった互換分岐は、残す明確な理由がない限り削除するべきである
    - シグネチャや型名から明らかな説明だけの docstring は短くするべきである

### realization code の追加は既存 realization code の整理と一体で行う

- 前提
    - コードベースの肥大化は、新しいファイル・関数・クラスを追加した時だけでなく、古い実装を残した時にも発生する
    - 既存実装の近くに同じ責務の実装が残ると、AI はどちらが正しいかを判断するために余計な文脈を読む必要がある
- 規則
    - realization code を追加する前に、同じ責務または近い責務を持つ既存の realization code を確認する
    - 既存実装の修正・移動・置換で現行仕様を満たせる場合は、新しい実装を追加しない
    - 新しい実装を追加した場合は、置き換えられた古い実装、不要になった分岐、重複した定数、古いテストも同時に削除または統合する
    - 一時的な移行コード・互換コードは、現行仕様から明示的に必要な場合だけ残す
    - 移行コード・互換コードを残す場合は、それを削除できる条件を近くのコメントに書く
- e.g.
    - 新しい parser を追加するなら、旧 parser を呼ぶ経路・旧 parser 専用テスト・旧 parser 専用 fixture も整理する
    - 既存関数に引数を 1 つ足せば足りる処理のために、ほぼ同じ関数を別名で作らない
    - `legacy_` や `old_` を名前に含む実装を残すなら、なぜ現行仕様で必要なのかを書き、不要なら削除する

### 新しい抽象化は実在する重複または明確な責務境界に基づく

- 前提
    - 抽象化はコード量を減らすこともあるが、呼び出し関係・引数・概念を増やしてコードベースを肥大化させることもある
    - 「将来使うかもしれない」抽象化は、現時点の AI の読解対象を増やす
- 規則
    - 新しい関数・クラス・モジュールは、現行仕様上の責務境界を明確にする場合、または実在する重複を削減する場合に作る
    - 単に処理が短いという理由だけで、1 箇所からしか呼ばれない小関数を増やさない
    - 共通化は、見た目が似ているだけではなく、入力・出力・失敗時挙動・不変条件が同じ場合に行う
    - 呼び出し元ごとの差分を大量のフラグや callback で吸収する汎用関数は避ける
    - 共有範囲が 1 モジュール内に閉じる場合は、そのモジュール内の private helper に留める
    - 複数のサブコマンドまたは複数の上位モジュールから使うことが明確な場合にだけ、共有モジュールへ移動する
- e.g.
    - 2 つの処理がどちらも git command を呼んでいても、失敗時の扱いが違うなら無理に 1 関数へまとめない
    - `do_everything(options: dict)` のような万能関数を作るより、責務が明確な小さな処理を呼び出し元の近くに置く
    - 1 箇所からしか呼ばれず、名前を付けても意図が増えない 3 行の処理は、無理に関数化しない

### 公開面・設定面・状態の増加を抑制する

- 前提
    - CLI 引数、サブコマンド、設定項目、環境変数、出力 schema、保存ファイルは、一度追加すると互換性維持の対象になりやすい
    - 公開面が増えると、実装・テスト・ドキュメント・利用者理解のすべてが増える
- 規則
    - 新しい CLI 引数、サブコマンド、設定項目、環境変数、出力項目、状態ファイルは、現行仕様を満たすために必要な場合だけ追加する
    - 内部実装の都合だけで公開面を増やさない
    - 単一の妥当な挙動を選べる場合は、設定項目で利用者に選ばせない
    - 新しい永続状態を追加する場合は、その生成条件・更新条件・削除条件を仕様または実装上で明確にする
    - 同じ情報をログ・状態ファイル・出力 schema に重複して保存しない
- e.g.
    - デバッグ用に一時的に欲しい値を、恒久的な CLI option として追加しない
    - 計算で再現できる値を、別ファイルに永続化しない
    - 仕様上 1 つの動作で十分なら、`--mode` や `--strategy` を追加しない

### realization test の肥大化も抑制する

- 前提
    - realization test も realization code の一部であり、読み取り・保守・更新の対象である
    - テストが重複すると、仕様変更時に実装よりテスト修正のほうが大きくなる
- 規則
    - テストは、実装詳細ではなく、現行仕様上意味のある外部挙動または制御ロジックを検証する
    - 同じ観点のテストケースは、可能な範囲で parametrized test や小さな fixture に集約する
    - 似たテストを追加する前に、既存テストへケース追加できないか確認する
    - 古い仕様に対応するテスト、削除済み実装に対応するテスト、同じ失敗を重複して検出するテストは削除または統合する
    - 大きな fixture ファイルは、現行仕様の検証に必要な最小内容にする
- e.g.
    - 同じ関数の正常系を入力値違いで 5 本のテスト関数に分けず、1 本の parametrized test にまとめる
    - 内部 helper の呼び出し順だけを固定するテストは、仕様上の出力や副作用を検証するテストに置き換える
    - Codex CLI や LLM の出力品質そのものを検証するためのテストを追加しない

### 依存関係・補助ファイル・生成物の増加を抑制する

- 前提
    - 外部依存、補助スクリプト、テンプレート、生成済みファイルは、実装本体以外にもメンテナンス対象を増やす
    - realization ancillary の増加も、AI が読むべき文脈と更新すべき対象を増やす
- 規則
    - 新しい外部依存は、標準ライブラリまたは既存依存で現行仕様を十分に満たせない場合だけ追加する
    - 外部依存を追加する場合は、それにより自前実装・テスト・保守負担がどれだけ減るかを重視する
    - 小さな処理のためだけに大きな依存を追加しない
    - 生成可能なキャッシュ、実行ログ、一時ファイル、ビルド成果物は realization files として持たない
    - 補助スクリプトを追加する場合は、既存の CLI・テスト・開発手順で代替できないか確認する
- e.g.
    - 数行の path 操作のために外部パッケージを追加しない
    - テストで必要な小さな入力は、巨大な fixture ファイルではなくテスト内で組み立てる
    - 実行すれば再生成できるファイルを、正本のようにコミットしない

### realization files の変更完了時には削除・統合余地を確認する

- 前提
    - 肥大化は、実装中よりも実装完了時に見落とされやすい
    - 「追加して動いた」状態は、「最小で保守しやすい」状態とは限らない
- 規則
    - realization files の変更完了前に、追加したもののうち削除・統合・短縮できるものがないか確認する
    - 未使用の関数、クラス、import、定数、fixture、補助ファイルを残さない
    - 同じ責務を持つ名前違いの実装を残さない
    - 現行仕様の説明に不要な TODO、NOTE、コメントを残さない
    - 変更後の realization files は、現行仕様を満たすための最小構成になっているべきである
- e.g.
    - 実装の途中で作った確認用 helper を、最終的に使わないなら削除する
    - 新しい共通関数へ移行した後、旧 helper の import とテストだけが残っていないか確認する
    - 仕様説明に寄与しない「念のため」コメントは削る
