
# oracles files 記述標準

## 前提

- oracles files の基本情報は `<cmoc-root>/oracles/docs/app_specs/oracles/basics.md` を参照すること
- oracles standards とは oracles files の内容が標準的に従うべき事項である
- 人間が oracles standards と守れるように、 cmoc はサブコマンド経由で人間を支援する
- `<cmoc-root>` 配下の oracles files それ自身もまた oracles standards に従わなければならない

## 記述標準

### oracles files はテキスト化された人間意図である

- 前提
    - エージェント AI は許可された情報にしかアクセス出来ない
    - 開発の対象 (`<repo-root>`) について責任を持つ主体は人間である
    - cmoc が従うべきなのは、究極的には人間意図である
- 規則
    - 人間の脳内にしかない情報・意図を oracles files に記述するべきである

### 人間の認知能力の消費は節約しなければならない

- 前提
    - 人間の認知能力は人間だけが提供できる希少なリソースである
    - 希少性ゆえに、あらゆるケースで人間の認知能力の供給不足ボトルネックになる
- 規則
    - 人間の認知能力は可能な限り節約しなければならない
    - 人間の認知能力の投入量あたりの成果量（＝コストパフォーマンス）の最大化を目指す
    - 最大限レバレッジをか効かせる事を目指す

### oracles files は正本仕様断片である

- 前提
    - oracles files は人間が 100% 責任を持つことになっている
    - oracles files の規模はそのまま人間のメンテナンスコストと比例することになる
    - 人間の認知負荷を下げるためには oracles files の規模は小さく保たなければならない
- 規則
    - oracles files は正本仕様の断片であると定義する
    - 正本仕様断片が疎な状態を積極的に目指す
    - 正本仕様の網羅的記述は避ける
- e.g.
    - 文言の過剰なカテゴリ分け

### 仕様断片の隙間の未定義動作は許容する

- 前提
    - oracles files 正本仕様断片である
    - 仕様断片の隙間には未定義動作が生まれるはずである
    - 未定義動作は、実装者 (AI) による実装差が生まれる原因となる
- 規則
    - 人間の認知負荷の節約を重視し、未定義動作の存在はむしろ歓迎する
    - oracles files 上に未定義動作が存在する場合、それは人間があえて是認しているとみなす
    - もしも実装差が許されないのなら、人間の責任であるべき実装（＝仕様）を oracles files として明示する
    - 実装者 (AI) が「正本仕様断片」「現在の既存実装・既存テスト」から「妥当な新実装・テスト」を導き出せる事を仮定して良い
    - 実装者 (AI) の裁量で未定義動作部分の仕様を定めて良いが、「AI 裁量で定めた仕様の規模」は小さい事が望ましい

### oracles files の総文字数の最小化を目標とする

- 前提
    - 人間の認知負荷を下げるためには oracles files の規模は小さく保たなければならない
- 規則
    - 必ず守らなければならない要件を満たしている範囲内（解空間内）で、 oracles files 全体で見た時の総文字数が最小となること（文字数最小解）を目指す
- e.g.
    - 複数箇所で全く同じ事を述べる必要があるのなら、１箇所に記述をまとめた上で、それを複数箇所で参照する
    - 文字数を減らすために冗長な言い回しは避けるべきだが、だからと言って助詞を省略してはいけない

### 正本仕様断片の間で論理的な矛盾はあってはいけない

- 前提
    - 実装は正本仕様断片と矛盾してはいけない
    - もしも、正本仕様断片が矛盾しているのであれば、実装も矛盾することになる
- 規則
    - 正本仕様断片の間で、解釈の余地のない明確な論理的矛盾はあってはいけない
- e.g.
    - 「基本原則」と「具体的な詳細仕様」とは、論理的に整合しなければいけない

### 実装から仕様への逆流は原則として認めない

- 前提
    - oracles files に記述された正本仕様断片を元に実装が生成される
    - 人間の意図が正確に直接反映されているのは oracles files だけである
- 規則
    - oracles files を調査せずに、実装だけから正本仕様を導き出すのは禁止
    - 実装から導き出された仕様を oracles に反映するのは原則として禁止
    - 「実装不可能な正本仕様」の調査を目的とする場合のみ、実装から正本仕様導出を導き出しても良い

### 用語・表記は統一されていなければならない

- 前提
    - 読むべきファイルを特定する方法として、キーワード検索は重要な手がかりである
- 規則
    - 表記揺れは統一しなければならない
    - 複数回登場する概念には専用の名前を割り当てるべきである
    - typo は修正しなければならない

### 命名は適切でなければいけない

- 前提
    - AI は命名から実際を推測しがちである
    - 名前と実態とが乖離していると、結果的に人間意図と実装とが乖離する
- 規則
    - 「名前から推測される説明」と「oracles files から読み取れる実際上の説明」とは整合しなければいけない

### ベストプラクティスよりも oracles files 記述標準で定義されている規則が優先される

- 前提
    - 一般的に良しとされていることが cmoc でも良しとされるかは別の問題である
- 規則
    - 「一般的に知られているベストプラクティス」と「oracles files 記述標準の規則」とが競合した場合は「oracles files 記述標準」を優先する
    - 「一般的に知られているベストプラクティス」よりも「oracles files 記述標準の原則」が優先される

### 仕様断片として goal だけではなく non-goal も書くことが望ましい

- 前提
    - 実装者 (AI) が正本仕様断片を解釈するうえで「そうである」と「そうではない」の境界は重要である
- 規則
    - 可能な限り「そうである (goal)」と対になる「そうではない (non-goal)」を記述するべきである
    - goal だけの記述も許容される
