
# oracles ファイル記述標準

## 前提

- oracles ファイルの基本的な定義については `<cmoc-root>/oracles/docs/app_specs/oracles/basic_definitions.md` を参照すること
- 基本的な定義を踏まえた上で、「oracles ファイル記述標準」は oracles ファイルについての標準的な原則・規則を述べる

## 原則

### oracles ファイル記述標準の立ち位置

- cmoc はこの標準に従って oracles ファイルを取り扱うことを想定する
- 人間がこの標準を守れるように cmoc は支援を行うことを想定する

### oracles ファイルはテキスト化された人間意図である

- cmoc による開発の対象 (`<repo-root>`) について責任を持つ主体 (i.e. 人間) が実現したい意図を書く
- 人間の脳内にしかない情報・意図を Codex CLI が利用可能な形式にする事を意味する

### 人間の認知能力の消費は節約しなければならない

- 人間の認知能力は人間だけが持ち出せる希少なリソースであるという立場を取る
- 希少性ゆえに、あらゆるケースで人間の認知能力の供給不足ボトルネックになる事を想定・前提とする
- よって、人間の認知能力は可能な限り節約しなければならない
- 言い換えれば…
    - 人間の認知能力の投入量あたりの成果量（＝コストパフォーマンス）の最大化を目指す
    - 最大限レバレッジをか効かせる事を目指す

## 規則

### oracles ファイルは正本仕様断片である

- 前提
    - oracles ファイルは人間が 100% 責任を持つことになっている
    - oracles ファイルの規模はそのまま人間のメンテナンスコストと比例することになる
    - 人間の認知負荷を下げるためには oracles ファイルの規模は小さく保たなければならない
- 規則
    - oracles ファイルは正本仕様の断片であると定義する
    - 仕様断片が疎な状態を積極的に目指す
    - 仕様の網羅的記述は避ける

### 仕様断片の隙間の未定義動作は許容する

- TODO
    - むしろ未定義動作は AI がどうにか出来る余地として歓迎する
    - 未定義動作っていうネーミングはちょっと違うかも

- 前提
    - oracles ファイル正本仕様断片である
    - 仕様断片の隙間には未定義動作が生まれるはずである
    - 未定義動作は、実装者 (AI) による実装差が生まれる原因となる
- 規則
    - 人間の認知負荷の節約を重視し、未定義動作の存在は許容する
    - oracles ファイル上に未定義動作が存在する場合、それは人間があえて是認しているとみなす
    - もしも実装差が許されないなら、あるべき実装（＝仕様）を明示するのは人間の責任である
    - 実装者 (AI) が「正本仕様断片」「現在の実装・テスト」から、妥当な実装を導き出せる事を仮定してよい
