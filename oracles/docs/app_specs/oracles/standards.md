# oracles files 記述標準

## 前提

- oracles files の基本情報は `<cmoc-root>/oracles/docs/app_specs/oracles/basics.md` を参照すること
- standards の記述形式は `<cmoc-root>/oracles/docs/dev_rules/standards_format.md` に従うこと
- oracles standards とは oracles files の内容が標準的に従うべき事項である
- 人間が oracles standards を守れるように、cmoc はサブコマンド経由で人間を支援する
- `<cmoc-root>` 配下の oracles files それ自身もまた oracles standards に従わなければならない

## 記述標準

### oracles files はテキスト化された人間意図である

- 対象
    - oracles files
- 背景
    - エージェント AI は許可された情報にしかアクセスできない
    - 開発の対象 (`<repo-root>`) について責任を持つ主体は人間である
    - cmoc が従うべきなのは、究極的には人間意図である
- 要求
    - [必須] 開発対象について人間が cmoc や Codex CLI に従わせたい意図を oracles files に記述する
    - [推奨] 人間の脳内にしかない情報・意図のうち、実装差や判断差を避けたいものを優先して oracles files に記述する
- 判定
    - 人間意図に依存する重要な判断が、実装者である AI の推測だけに委ねられていない
    - oracles files を読めば、少なくとも人間が明示したい意図を追跡できる
- 例示
    - OK: 失敗時に安全側へ倒すべき条件を oracles files に明示する
    - NG: 人間だけが知っている制約を、既存実装から AI が推測することに任せる

### 人間の認知能力の消費は節約しなければならない

- 対象
    - oracles files
    - oracles standards
    - cmoc による oracles files 支援機能
- 背景
    - 人間の認知能力は人間だけが提供できる希少なリソースである
    - 希少性ゆえに、あらゆるケースで人間の認知能力の供給不足がボトルネックになる
- 要求
    - [必須] oracles files の記述・構成・量は、人間の認知能力の消費を節約する方向で決める
    - [推奨] 人間の認知能力の投入量あたりの成果量を最大化する
    - [推奨] 人間が少ない判断で大きな実装成果を得られるように、oracles files のレバレッジを高める
- 判定
    - 人間が読み直すべき文脈量が不要に増えていない
    - 人間が判断すべき事項と AI に任せてよい事項の境界が読み取れる
- 例示
    - OK: 人間判断が必要な境界条件だけを oracles files に書き、細部の実装選択は AI に任せる
    - NG: 人間判断が不要な実装手順を網羅的に列挙する

### oracles files は正本仕様断片である

- 対象
    - oracles files
- 背景
    - oracles files は人間が 100% 責任を持つ
    - oracles files の規模は人間のメンテナンスコストに影響する
    - 人間の認知負荷を下げるためには、oracles files の規模を小さく保つ必要がある
- 要求
    - [必須] oracles files を正本仕様の断片として扱う
    - [推奨] 正本仕様断片が疎な状態を目指す
    - [禁止] 正本仕様を網羅的に記述すること自体を目的として oracles files を書いてはいけない
- 判定
    - oracles files が、実装差を避けたい重要事項を中心に書かれている
    - 仕様全体を埋め尽くすためだけの分類・列挙・説明が増えていない
- 例示
    - OK: 実装差が出ると困る入出力契約だけを正本仕様断片として書く
    - NG: 必要性の薄い文言を過剰にカテゴリ分けして網羅的に記述する

### 仕様断片の隙間の未定義部分は許容する

- 対象
    - oracles files
    - oracles files を解釈する実装者
- 背景
    - oracles files は正本仕様断片である
    - 仕様断片の隙間には未定義部分が生まれる
    - 未定義部分は、実装者である AI による実装差が生まれる原因となる
- 要求
    - [許容] oracles files 上に未定義部分が存在することを許容する
    - [必須] oracles files 上に未定義部分が存在する場合、それは人間があえて是認しているものとみなす
    - [必須] 実装差が許されない事項は、人間の責任で oracles files に明示する
    - [許容] 実装者である AI は、「正本仕様断片」「現在の既存実装・既存テスト」から妥当な新実装・テストを導き出してよい
    - [推奨] 実装者である AI の裁量で補う仕様の規模は小さく保つ
- 判定
    - 実装差が許されない事項が oracles files に明示されている
    - AI 裁量で補われる部分が、現行の正本仕様断片と既存実装・既存テストから自然に導ける範囲に収まっている
    - 未定義部分を埋めるためだけの過剰な仕様記述が追加されていない
- 例示
    - OK: エラー終了すべき条件だけを明示し、内部実装の関数分割は AI の裁量に任せる
    - NG: 実装差が許されない出力形式を oracles files に書かず、既存実装から推測させる

### oracles files の総文字数の最小化を目標とする

- 対象
    - oracles files
- 背景
    - oracles files の規模が大きいほど、人間の認知負荷が増える
    - oracles files の規模が大きいほど、AI が読む文脈量も増える
- 要求
    - [推奨] 必ず守らなければならない要件を満たしている範囲内で、oracles files 全体の総文字数が最小となることを目指す
    - [推奨] 複数箇所で同じ内容を述べる必要がある場合は、1 箇所にまとめたうえで参照する
    - [禁止] 文字数削減のために意味が曖昧になる省略をしてはいけない
- 判定
    - 同じ意味の記述が複数箇所に重複していない
    - 冗長な言い回しが残っていない
    - 助詞や接続関係の省略により解釈が不安定になっていない
- 例示
    - OK: 複数箇所で同じ説明を書く代わりに、1 箇所にまとめて参照する
    - NG: 文字数を減らすために助詞を削り、文の係り受けを曖昧にする

### 正本仕様断片の間で論理的な矛盾はあってはいけない

- 対象
    - oracles files
    - 正本仕様断片
- 背景
    - 実装は正本仕様断片と矛盾してはいけない
    - 正本仕様断片が矛盾している場合、実装も矛盾する
- 要求
    - [禁止] 正本仕様断片の間に、解釈の余地のない明確な論理的矛盾があってはいけない
    - [必須] 基本原則と具体的な詳細仕様は論理的に整合していなければならない
- 判定
    - ある正本仕様断片に従うと、別の正本仕様断片に必ず違反する状態になっていない
    - 一般的な方針と個別仕様の優先関係が読み取れる
- 例示
    - OK: 基本原則と具体的な詳細仕様が同じ結論を導く
    - NG: ある仕様では「必ず実行する」と書き、別の仕様では同じ条件で「実行してはいけない」と書く

### 実装から仕様への逆流は原則として認めない

- 対象
    - oracles files
    - oracles files の作成・修正提案
- 背景
    - oracles files に記述された正本仕様断片を元に実装が生成される
    - 人間の意図が正確に直接反映されているのは oracles files である
    - 実装には、AI の裁量・過去の都合・偶然の挙動が含まれ得る
- 要求
    - [禁止] oracles files を調査せずに、実装だけから正本仕様を導き出してはいけない
    - [禁止] 実装から導き出された仕様を、正本仕様としてそのまま oracles files に反映してはいけない
    - [許容] 実装不可能な正本仕様の調査を目的とする場合のみ、実装から見える制約や矛盾を oracles files の修正提案材料として扱ってよい
- 判定
    - oracles files の変更提案が、既存実装の都合だけを根拠にしていない
    - 実装から見える制約を扱う場合、それが正本仕様の問題調査として位置づけられている
- 例示
    - OK: oracles files と既存実装が矛盾している箇所を見つけ、人間判断が必要な修正候補として提示する
    - NG: 既存実装がそうなっているという理由だけで、oracles files に同じ挙動を正本仕様として追記する

### 用語・表記は統一されていなければならない

- 対象
    - oracles files
    - oracles files 内の用語・表記
- 背景
    - 読むべきファイルを特定する方法として、キーワード検索は重要な手がかりである
    - 同じ概念に複数の表記があると、人間と AI の検索・読解が不安定になる
- 要求
    - [必須] 同じ概念を指す表記揺れは統一する
    - [推奨] 複数回登場する概念には専用の名前を割り当てる
    - [必須] typo は修正する
- 判定
    - 同じ概念が複数の名前で呼ばれていない
    - キーワード検索で関連する正本仕様断片を見つけられる
    - typo により別概念のように見える表記が残っていない
- 例示
    - OK: 同じ概念を常に `oracles files` と表記する
    - NG: 同じ概念を `oracle files`, `oracles file`, `oracles files` と揺らして書く

### 命名は適切でなければいけない

- 対象
    - oracles files
    - oracles files 内で定義・使用される名前
- 背景
    - AI は命名から実態を推測しやすい
    - 名前と実態が乖離していると、人間意図と実装が乖離しやすい
- 要求
    - [必須] 名前から推測される説明と、oracles files から読み取れる実際上の説明は整合していなければならない
    - [禁止] 実態と異なる推測を強く誘発する名前を使ってはいけない
- 判定
    - 名前だけを見たときの自然な解釈が、oracles files 上の定義と矛盾していない
    - 既存用語との関係が読み取りにくい名前になっていない
- 例示
    - OK: 一時実行用の branch に、永続 branch と誤解されない名前を付ける
    - NG: 削除される一時ファイルに `permanent` を含む名前を付ける

### ベストプラクティスよりも oracles standards で定義されている要求が優先される

- 対象
    - oracles files
    - oracles standards の解釈
    - cmoc による oracles files 支援機能
- 背景
    - 一般的に良しとされていることが cmoc でも良しとされるとは限らない
    - cmoc は oracles files を正本仕様断片として扱う
- 要求
    - [必須] 一般的に知られているベストプラクティスと oracles standards の要求が競合した場合は、oracles standards を優先する
    - [必須] 一般的に知られているベストプラクティスを理由に、oracles standards の要求を無視してはいけない
- 判定
    - 判断の根拠が、一般論ではなく oracles standards と正本仕様断片に置かれている
    - ベストプラクティスを採用する場合でも、oracles standards と矛盾していない
- 例示
    - OK: 一般には網羅的な仕様書が望ましい場面でも、oracles standards が疎な正本仕様断片を求めるなら疎に保つ
    - NG: 一般的なドキュメントベストプラクティスを理由に、oracles files を網羅的に書き直す

### 仕様断片として goal だけではなく non-goal も書くことが望ましい

- 対象
    - oracles files
    - 正本仕様断片
- 背景
    - 実装者である AI が正本仕様断片を解釈するうえで、「そうである」と「そうではない」の境界は重要である
    - non-goal がない場合、AI が人間意図を越えて仕様を広げることがある
- 要求
    - [推奨] 可能な限り、goal と対になる non-goal を記述する
    - [許容] goal だけの記述も許容する
- 判定
    - 実装者である AI が、実装すべきことと実装しなくてよいことの境界を読み取れる
    - non-goal を書かない場合でも、過剰な実装を誘発しにくい記述になっている
- 例示
    - OK: `cmoc review oracles` が検出する対象と、検出しない対象を対で書く
    - NG: 目的だけを書き、明らかに対象外にしたい処理まで AI が追加し得る状態にする
