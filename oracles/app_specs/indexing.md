
# `cmoc` における `<repo-root>` 上の `INDEX.md` の扱い

## 概要

- `<repo-root>` 配下の大半のディレクトリ（後述）には `cmoc` によって `INDEX.md` が配置される
- この `INDEX.md` 配置は `cmoc` によって自動的にメンテナンスされる（人間は書かなくて良い）

## `INDEX.md` 配置対象ディレクトリ

- `<repo-root>` 配下のディレクトリを再帰的に列挙する
- 以下のディレクトリは配置対象から除外する
    - `.` から始まるディレクトリ
    - `.gitignore` の対象
    - `<repo-root>/memo`
    - `build`, `tmp` などのいかにもな名前のディレクトリ
    - バイナリファイル

## 目次作成対象ファイル・ディレクトリ

- `INDEX.md` と同階層のファイル・ディレクトリを列挙する
- 以下のファイル・ディレクトリは目次作成対象から除外する
    - `INDEX.md` 自身
    - `.` から始まるファイル・ディレクトリ
    - `.gitignore` の対象
    - バイナリファイル

## `INDEX.md` のフォーマット

- ファイル・ディレクトリ１件ごとに、以下のフォーマットの「目次情報」を用意する
    ```markdown
    # `<ファイル・ディレクトリ名>`

    ## Summary

    <ルーティング先ファイル・ディレクトリの説明>

    ## Read this when

    <どんな作業・質問・変更のときに、このファイル・ディレクトリを読むべきか>

    ## Do not read this when

    <逆に、このファイル・ディレクトリを読まなくて良い・読んではいけないケース>

    ## hash

    <この目次情報を生成するにあたって参照したファイル・ディレクトリとそのハッシュ>
    ```
- `<ルーティング先ファイル・ディレクトリの説明>`
    - ファイル・ディレクトリに何が書いてあるか・入っているかを書く
    - 箇条書き
- `<どんな作業・質問・変更のときに、このファイル・ディレクトリを読むべきか>`
    - AI がこれを読む判断を下す「条件」を明示する
    - 箇条書き
- `<逆に、このファイル・ディレクトリを読まなくて良い・読んではいけないケース>`
    - AI が「関連しそう」程度で過剰に読みに行く傾向を抑制するための「そうじゃないよ」を書く
    - 箇条書き
- `<この目次情報を生成するにあたって参照したファイル・ディレクトリとそのハッシュ>`
    - ファイルの場合は単にそのファイルの文字列（中身）から計算したハッシュ値を書く
    - ディレクトリの場合は「そのディレクトリの直下の目次作成対象ファイル・ディレクトリのハッシュ値を全結合した文字列」から計算したハッシュ値を書く
    - 箇条書き

## メンテナンス処理の仕様

### 処理対象

1. 配置対象ディレクトリは前述のルールで機械的に列挙する
2. 列挙されたディレクトリを「深い方から」順番に処理する
    - e.g. `<repo-root>/parent/INDEX.md` で `<repo-root>/parent/child` の目次情報を作成するために `<repo-root>/parent/child/INDEX.md` を閲覧する可能性がある
    - よって、子ディレクトリ側を優先して新鮮な状態にする
3. メンテナンス処理によって発生した git 未コミット差分を自動コミット

### ディレクトリ１つごとの処理

- `INDEX.md` が存在しなければ新規に作成
- `INDEX.md` 内の目次情報の不備をチェック
    - ファイルシステム上存在するファイル・ディレクトリと対応する目次情報が存在しなければ新規に作成する
    - 存在しないファイル・ディレクトリへの目次情報は削除
    - 目次情報に記録されているハッシュ値と実際のハッシュ値が一致しない場合は目次情報を削除・再生成

### 目次情報の生成方法

- Codex CLI に目次情報の生成を依頼する
- `codex exec` の呼び出しは、目次ファイルの生成対象１つごとに独立に行う
- 「ファイル・ディレクトリ名」「ハッシュ値」などの機械的に処理可能な部分は `cmoc` が責任を持つ（Codex CLI にやらせない）
- 目次情報は Structured Output で受け取る
- Structured Output の schema は以下の通り
    ```json
    {
        "type": "object",
        "additionalProperties": false,
        "required": ["summary", "read_this_when", "do_not_read_this_when"],
        "properties": {
        "summary": {
            "type": "array",
            "items": { "type": "string" }
        },
        "read_this_when": {
            "type": "array",
            "items": { "type": "string" }
        },
        "do_not_read_this_when": {
            "type": "array",
            "items": { "type": "string" }
        }
        }
    }
    ```

## メンテナンス処理の実行タイミング

- Codex CLI を実行する前に必ず実行する
- ただし「`INDEX.md` を生成するための Codex CLI 呼び出し」は対象から除外する
- e.g. `eval-oracles` で Codex CLI による調査作業を行う前に `INDEX.md` メンテナンスを実行する

## `INDEX.md` が最新に保たている状態でメンテンナス処理が実行されたら

- 機械的チェックによって、最新状態であることが確認される
- その結果 Codex CLI による目次生成は一切実行されない
