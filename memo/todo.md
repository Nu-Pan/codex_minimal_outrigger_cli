## branch, apply, merge 大工事の余波

- 大工事の過程で、未コミット差分なしだと通らないようになったけど、これは戻したい
- ブランチのメタ名は `<cmoc-...>` みたいにプリフィックス cmoc を付ける方向性で統一したい
- 意図がよくわからん
    > apply 開始後に `<cmoc-session-branch>` へ追加された commit は、実行中の apply の部分適用対象には含めない。
- `<cmoc-apply-branch>` の `<cmoc-session-branch>` へのマージ作業はサブコマンドとして導入したい
    - `cmoc apply fork`, `cmoc apply join` とかだろうか
    - であれば `<cmoc-session-branch>` の方は `cmoc session fork`, `cmoc session join` の方が良いだろう
    - git の branch, merge とは別概念なので、別ワードを割り当てたいという意図もある
- `cmoc session report` が必要
    - 今この瞬間に `cmoc session join` したら、どういう内容になるか？　をレポートさせる
    - このまま `cmoc session join` に進んでよいかの判断材料
- スキーマをどこにも書いてない
    - `<repo-root>/.cmoc/sessions/<session-id>/session.json`
    - `<repo-root>/.cmoc/sessions/<session-id>/apply-runs/<apply-run-id>.json`
- oracles として用語集が必要
    - 用語関係は将来的には cmoc の機能にしたいが、直近は oracles に手作業で書く
- どこに書くのん
    > `<cmoc-apply-branch>` を `<cmoc-session-branch>` に取り込む操作は、`cmoc merge` の責務ではない。
- `<cmoc-apply-branch>` の削除ルール
    - 書いてない

## cmoc が作業している間も oracles を編集したい

- 今の仕様だと cmoc の作業中に oracles を触るとエラーで止まる
- 流石にしんどいのでなんとかしたい

## git 分岐・マージ周りの仕様を再整理したい

- master へのマージではなく、分岐元ブランチへのマージとしたい
- どのみち、 apply で専用ブランチを作成することを考えるとまとめて再検討が必要という話ではある

# cmoc branch が未コミット差分ありでも通る仕様に戻したい

- 普通に不便なので、未コミット差分は許容したい

## Codex CLI の呼び出しを可能な限り並列にしたい

- トークンリミットで律速される想定になったので、コマンド１つが返ってくるまでの時間は短いに越したことはない
- R/O の評価系処理は並列実行が可能
- `INDEX.md` の生成も、依存関係グラフの末端側から解決するルールとすることで、依存関係が存在しないところは並列処理可能なはず

# Codex CLI 更新チェック

- 最大で１日１回の自動更新処理は cmoc に組み込んでおきたい

# `<cmoc-branch>` をヒントとして渡す

- 「今の作業と関係のある変更は `<cmoc-branch>` に乗ってるからそれ見てね」を各コマンドでやりたい
- 主に「どこ見ていいか分からないから全チェック」を回避するのに使いたい
- 具体的にどういう注入のしかたするのかは gpt と相談

# 読むべきドキュメントの候補を機械的に列挙してプロンプトに注入する

- 今は、何を読むかは AI 任せになっているが、この判定処理にもトークンが消費される
- ワード検索などで機械的に列挙してファイルリストをプロンプトに注入することで、トークン消費の緩和を狙う
- 過去の Codex CLI 呼び出し履歴から、実際に行われているテクニックを抽出する
- 多分、あえてコマンドの実行結果（検索コマンドとか）をそのまま流し込んだ方が AI には伝わる

# テスト実行パス

- 汚しても良いところでトイプログラムで実行ということにする
- パスとかトイプログラムの仕様とかを決めないといけない

# `eval-oracles` で読むべきファイルのリストをより正確にリストアップしたい

- `INDEX.md` とは別の、依存関係グラフが必要
- 依存関係情報を `INDEX.md` に書き込むのもアリかもしれない（see also として役に立つはず）
- 「意味的に関係があるか？」の判断のコアは Codex CLI になるが、機械的に処理可能な部分の責任は cmoc が持ち、 Codex CLI が担う責任範囲は最小化する

# バグ修正は正式なワークフローとして組み込んでも良いかも

- バグ修正は状態遷移的な操作なので、既存の oracles - apply ループには乗せにくい
- `oracles/bugs` とかに過去のやらかし事例を集積し、それらと同じ轍を踏んでいない事を apply の過程でチェック、とかだろうか

# `eval-oracles` に用語統一機能を組み込みたい

- 用語リストに基づいたファイル別チェックを駆使すれば、かなり安く行けるはず。
- 修正が漏れても良いタイプなので、 reasoning effort は低くても大丈夫なはず

# 無茶な仕様を調べる方法が欲しい

- 今は `oracles` にかかれている仕様が絶対視されている
- `oracles` の仕様の品質は AI レビュー＋人間修正のループによって保たれる
- 問題は、このレビューの際に「実装は見てはいけない」としている点
- 「言われた通りに頑張って実装したけど、やっぱこの仕様おかしいッスよ」を拾えるフローがどこにもない
- `eval-oracles` の機能として組み込みたい

## 使用モデル設定をプロファイル由来にしたい

- 今はモデルと Reasoning Effort を引数で直接指定している
- これだと、モデルセットが更新された時に cmoc 側の追従が必要でかなりダルい
- なので `<repo-root>` 側の設定として cmoc 用のプロファイルを書いてもらうことにする
- `cmoc dector` サブコマンドが必要だろう

## 「仕様に書いてないことが実装されている」をどう扱うか

- 例えば、単純に oracles ファイルを削除したりすれば「実装は存在するが、そのことは仕様には書いてない」は簡単に起きうる

## README.md も AI によるメンテの対象にしたい

- README.md の仕様を oracles に書けば、適切な README を書いてくれるはず

## HQ モードが欲しい

- 今の仕様はトークン効率を重視した設定になっている
- 要所要所で高性能モデルを発見しているので False-Positive は防げるが、要修正点の Recall は何ら保証されていない
- これを解決するには、要修正点列挙作業の設定を重たいものにするしか無い
- ということで、全ての Codex CLI 呼び出しの最低ラインをフロンティアモデルの Reasoning Effort = medium にするモードを追加したい

# 補助的なサブコマンド

## `cmoc commit` コマンドが欲しい

- 未コミット差分をコミットするだけ
- Codex CLI を使ってコミットメッセージを自動生成してくれる

## `cmoc status` コマンドが欲しい

- 現在のブランチが cmoc branch か
- base commit は何か
- 未コミット差分が oracle 内だけか
- 最後の eval-oracles / apply レポートはどれか
- apply が完了扱いか未完了扱いか
- merge 可能そうか

## `cmoc dector` コマンドが欲しい

- codex が呼べるか
- Structured Output が使えるか
- .cmoc が gitignore されているか
- oracles/INDEX.md があるか
- 必要な git 状態か

## `cmoc dev kaizen` サブコマンド

- cmoc 自体を改善するための情報を収集してレポートする機能
- 本番運用上でのフィードバックループを回すための機能
- 必要な情報は `<repor-root>` 上でしか収集出来ないので、 cmoc のサブコマンドに組み込む事自体は正しい方向性なはず
