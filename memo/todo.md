## 各プロンプトをきっちり仕様化

- 今は指定が雑なので AI エージェントの仕事も雑

## cmoc が生成するプロンプトと、リポジトリ固有指示との整合性チェックが欲しい

- 「リポジトリ側の `AGENTS.md` なり oracle ファイルなりで書いた指示」と「cmoc のプロンプト由来の指示」とが矛盾する可能性は当然ある

- `cmoc review oracle` とかだろうか？

## oracle file の隙間を埋められる根拠を言わせるべき

- 具体的にどういう隙間があって、それをどう埋められるか、あるいは埋められないか
- これを述べさせないと、適当にダイジョウブダヨって言いそう

## コメントをめっちゃ書かせたい

- コメントはめっちゃいっぱい書くべき
    - コードから対応する oracle にジャンプできるかどうかはコメントにかかっている
    - 適切な対応 oracle に飛べなくてナンセンスなコードを生成しまくって巨大コードベース化するとトークン消費がエグいことになる
    - ナンセンスなコード生成はコントロール不可能な肥大化につながるが、コメントの詳細化によるコードベース増大はコントロール可能
- docstring とか、コードブロックごとのコメントとか、積極的に書くように realization standard に追記したい
- 中身としては
    - コードだけからは読み取れない実装意図
    - この実装でなければならない根拠
    - あえて取らなかった実装方針とその根拠
    - 対応する oracle file が存在するなら、そのファイルパス (`<work-root>` 起点で)

## ファイル別をやる前に、差分ベースの全体 apply をやったほうが良いかも

- リポジトリ全体に対する「oracle file にこういう修正いれたから追従させて」を先にやる
    - 一発でうまくいくわけ無いので、所見リストアップ --> 実装を繰り返し実行
    - もう所見ないよって言われるまで繰り返し
    - その後で、仕上げにファイル単位の網羅的チェックを行う
- 気持ち
    - ファイル単位調査だと、ファイル個別の事情に釣られやすそう
    なので「抜け漏れは有るかもしれないが、大筋対応出来ている」な状態を先に作る
    - その後であれば、ファイル単位修正が多少近視眼的でも大きな問題にはならないはず
    - 実際、人力での開発の時も、大抵はそんな流れのはず
- せっかく GPT-5.6 が来るので
    - Sol による自動オーケストレーションを試しても良いかも

## prompt parts を最適化する

- 意味的な重複がかなり多い
- 入力トークン数がかさむし、指示の実効性にも不安があるので、重複を整理して短縮したい

## プロファイルベースでプロンプトブロックの順序を最適化する

- 今の順序が最適であるという確認はしていない
- 実際の過去の呼び出しパターンを元に順序を最適化したほうが良い
- シミュレーションは可能なはず
- 頻繁に変えるとキャッシュが効かなくなるので、サブコマンドで再計算とかだろうか

## 所見リストの編集をファイル経由にする

- 今は、編集前の所見リストを json でプロンプトに注入し、編集結果あるいは編集操作を Structured Output で受け取る形式にしている
- これをファイル経由にすれば、普通のソースコードの編集能力を活かせるので、所見リスト編集作業を無理無くこなせるはず
- その agent call で変化があったかも、ファイルのハッシュ値を見れば一目瞭然
- スキーマとそれを用いた機械的チェックの手段さえ与えればかなり有効なはず
- 読み書き規則のパターンが増えるかも

## 所見リストのマージと改善は別枠にしたほうが良い

- 改善作業には高価な高性能モデルが必要だが、高性能モデルは本当に高い
- 先に中性能モデルで所見リストをコンパクションして高性能モデルに渡す、高性能モデルによる改善は１～２周だけ、みたいな削減施策が必要

- quota poll の消費トークン数を削減
    - 多分 AGENTS.md とかを読みに行っていて、無駄に消費している

## サブコマンド実行の所要時間を書き分けたい

- 並列処理するようになったので、エージェント実行時間の総和（もしも直列実行してたらの総時間）も欲しい

## 作業の停止性を良くしたい

- 反復オーケストレーションは停止性っぽい問題を抱えている
- e.g. Codex CLI が消化した以上に Codex CLI 自身が仕事を作る --> 反復のハードリミットいっぱいまで作業
- e.g. イテレーションを跨いで編集合戦を行う --> 循環参照的な構造が発生、反復のハードリミットいっぱいまで作業
- つまり、ハードリミットに達した時に、本当に作業が残っているのか、無駄な作業を無駄に作りまくっているのか、この区別が付かない
- この不安を払拭するために、工夫を入れたい
- 問題たりえるのは「Codex CLI が生成する仕事量」だけなので、ここを減らすしか無い

## `INDEX.md`

- トークン消費・実行時間両面で問題を抱えている
- `INDEX.md` のメンテナンスにかかるトークン消費が意外とエグそう
    - chached input が効いているので、実は大したコストではない説はある
- 修正量の多さと意味的な重要さは必ずしも比例しない
    - e.g. typo が忠実に実装されてしまっていたので修正した --> たとえ１文字でも確実に `INDEX.md` に
- 量的に僅かな修正であっても意味的な重要な修正である可能性を考えると、全体を再生成せざるを得ないというのが厳しい
- 非エージェントの検索システムが必要なのでは
- そこまでやり始めたらもはや minimal ではないので、派生した別プロダクトかも
- `INDEX.md` を完全に並列にしたい
    - 以下のルールでいけそう？
        - `INDEX.md` を生成するのに `INDEX.md` を読んではいけない
        - ハッシュ値の計算に `INDEX.md` を含めてはいけない
- ファイル・ディレクトリで性質が違うかも
    - そのファイル単体を見ただけでは、意味のあるルーティング情報は作れないのでは？
    - リポジトリの全体像を理解していないと、正しいルーティング情報は作れないのでは？
- ルーティングでは根本的に不十分
    - 「ルーティングに従って関係のある情報を見つけたからこれでOK」みたいな感じで調査をサボる可能性がある
    - 多分「oracle file の表記揺れを統一してキーワード検索で関連文章を当てられるようにする」とか「see also を AI に生成させる」とかの、泥臭い事をちゃんとやるのが近道
    - cmoc に RAG システムを組み込むみたいな話はその後でやるべき
- 依存先ファイルは複数必要
    - そのエントリの言及対象だけでなく、関係するファイルも列挙する必要がある
    - 正確には、そのエントリのルーティング情報を書くにあたって参照したファイルすべて
    - じゃないと「読みに行った他のファイル」が原因でエントリが陳腐化したケースを拾えない
- 先に陳腐化したルーティング情報を削除するのが良いかも
    - 機械的処理で陳腐化したルーティング情報を全削除してから、ルーティング情報の再生成に進む
    - そうすれば、ルーティング情報の生成用に読んだ `INDEX.md` に陳腐化した情報が含まれてて、陳腐化した情報が伝播してしまうパターンを防止できる
    - ＋INDEX.mdは参照しても良い＋末端側から順番に
- 難易度に応じて手段を変えるという手もある
    - MINIMUM, EFFICIENCY を使い分けるとか
    - 機械的に生成するとか

- 必要な情報は `<repor-root>` 上でしか収集出来ないので、 cmoc のサブコマンドに組み込む事自体は正しい方向性なはず
