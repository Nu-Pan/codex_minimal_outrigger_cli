# 実行時間とトークン消費

## 作業の停止性を良くしたい

- 反復オーケストレーションは停止性っぽい問題を抱えている
- e.g. Codex CLI が消化した以上に Codex CLI 自身が仕事を作る --> 反復のハードリミットいっぱいまで作業
- e.g. イテレーションを跨いで編集合戦を行う --> 循環参照的な構造が発生、反復のハードリミットいっぱいまで作業
- つまり、ハードリミットに達した時に、本当に作業が残っているのか、無駄な作業を無駄に作りまくっているのか、この区別が付かない
- この不安を払拭するために、工夫を入れたい
- 問題たりえるのは「Codex CLI が生成する仕事量」だけなので、ここを減らすしか無い

## 「良かれと思って」を抑制したい

- oracles・実装が膨らむ事＝トークン消費の増加なので、出来るだけ小さく保ちたい
- できれば、 `cmoc apply fork` の段階から、コード膨らみにくいように誘導したい
- そのためには「良かれと思って」系をしないように抑制するような工夫が必要
- 単にプロンプトで指示するだけはなく、Structured Output でわざと「良かれと思って」を吐き出させて、それを無視するのが良いだろう

## デカい oracles ファイルを分割したい

- デカい oracles ファイルは調査対象が多いので、それに対する Codex CLI 呼び出しの負担が増えて、調査結果の抜け・漏れが懸念される
- 大きすぎるファイルの分割は `cmoc review oracles` で面倒を見たい
- しかし、大きすぎるってどのくらい？
    - schema 定義は独立したファイルにしておこう、くらいは思いつく
    - 大きさじゃなくて、意味論を基準にしたほうが良いかも

## `INDEX.md` やめたい

- `INDEX.md` のメンテナンスにかかるトークン消費が意外とエグそう
- 修正量の多さと意味的な重要さは必ずしも比例しない
    - e.g. typo が忠実に実装されてしまっていたので修正した --> たとえ１文字でも確実に `INDEX.md` に
- 量的に僅かな修正であっても意味的な重要な修正である可能性を考えると、全体を再生成せざるを得ないというのが厳しい
- 非エージェントの検索システムが必要なのでは
- そこまでやり始めたらもはや minimal ではないので、派生した別プロダクトかも

## `review oracles` に用語統一機能を組み込みたい

- 用語リストに基づいたファイル別チェックを駆使すれば、かなり安く行けるはず。
- 修正が漏れても良いタイプなので、 reasoning effort は低くても大丈夫なはず
- oracles の規模が大きくなってきて、用語統一が大変だと感じるようになってきたので、早めに欲しい
- あと、 AI に提供するインデクシング情報としても有用なはず
    - ワードの統一が保証されているのであれば、検索ベースの関連文章検索の信頼度が上がる

## `review oracles` で読むべきファイルのリストをより正確にリストアップしたい

- `INDEX.md` とは別の、依存関係グラフが必要
- 依存関係情報を `INDEX.md` に書き込むのもアリかもしれない（see also として役に立つはず）
- 「意味的に関係があるか？」の判断のコアは Codex CLI になるが、機械的に処理可能な部分の責任は cmoc が持ち、 Codex CLI が担う責任範囲は最小化する

## 読むべきドキュメントの候補を機械的に列挙してプロンプトに注入する

- 今は、何を読むかは AI 任せになっているが、この判定処理にもトークンが消費される
- ワード検索などで機械的に列挙してファイルリストをプロンプトに注入することで、トークン消費の緩和を狙う
- 過去の Codex CLI 呼び出し履歴から、実際に行われているテクニックを抽出する
- 多分、あえてコマンドの実行結果（検索コマンドとか）をそのまま流し込んだ方が AI には伝わる

## トークン消費の傾向を分析出来るようにしたい

- 分析に必要な情報を全部ログに出すようにする
- やりたいこと
    - そのサブコマンドの実行にかかった費用 (USD ベース) を試算
    - どの処理にどのくらい費用を食われているかをプロファイリング

# 補助的なサブコマンド

## `cmoc eval session`

- `<cmoc-session-branch>` 上で行った変更を全て総合したら、結局どういうことなのか？　をレポートさせる
- おかしなことが起きていないかに気づくための判断材料

## `cmoc propose oracles`

- 仕様変更を自由記述して、それに対して必要な作業を提案してくれる
    - 「とりあえず書きました」みたいなのが多すぎる
    - 文章設計みたいな概念のない、メンテがしんどい感じの文章を平気で出してくる
    - だからソースコードのレイヤーでは AI 生成物と付き合わないことにしたという話なんだけど
    - 仕様のドラフトを AI に書かせるにしても、もうちょっとマシな品質のものを出してほしい
- 低品質な oracles を提案しないように
    - プロンプトをガッツリ組む
    - オーケストレーションをガッツリ組む
- 仕様書のコーディング規約が必要
    - 良い書き方、悪い書き方、そういうのを事細かに oracles に明示する必要がある

## `cmoc index` サブコマンド

- `INDEX.md` の構築だけを行うコマンド
- `INDEX.md` を最新に追従させてから Codex CLI を直接呼び出してなんかする…みたいな用途を想定

## `cmoc commit` サブコマンド

- 未コミット差分をコミットするだけ
- Codex CLI を使ってコミットメッセージを自動生成してくれる
- ただのヘルパー関数
- ついでなので cmoc が生成するコミットメッセージも規約化したほうが良いか

## `cmoc status` サブコマンド

- 現在のブランチが cmoc branch か
- base commit は何か
- 未コミット差分が oracle 内だけか
- 最後の review oracles / apply レポートはどれか
- apply が完了扱いか未完了扱いか
- merge 可能そうか

## `cmoc dector` サブコマンド

- codex が呼べるか
- Structured Output が使えるか
- .cmoc が gitignore されているか
- oracles/INDEX.md があるか
- 必要な git 状態か

## `cmoc kaizen fork` サブコマンド

- cmoc 自体を改善するための情報を収集してレポートする機能
- 本番運用上でのフィードバックループを回すための機能
- 必要な情報は `<repor-root>` 上でしか収集出来ないので、 cmoc のサブコマンドに組み込む事自体は正しい方向性なはず
