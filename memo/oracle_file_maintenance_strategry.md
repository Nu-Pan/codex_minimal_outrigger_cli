
# oracle file メンテナンス戦略

## 前提

- oracle file は人間が積極的に関与する唯一の領域
- AI agent によるレバレッジを大きくするには oracle file のメンテナンスの手間の削減は必須

## ブレスト

### 人間の監視は必要？
 100% AI 任せだと、ナンセンスな事をし始めてめちゃくちゃになる？
- でも、誤字脱字の修正みたいなシンプルな話は AI 任せにしてラクしたい
- 「この仕様で良いのか？」とか「この仕様文言でいいのか」みたいな高度な判断は人間で握りたい
- ドラフトくらいまでなら AI にやらせても良いかも
- 修正方法の提案も AI にやらせても良い
- AI による oracle file の直接編集は原則禁止で、誤字脱字みたいな単純修正は例外的に許可、みたいな感じか

### cmoc のサブコマンド構成は？

- 「AI にターンを回す」みたいな役割の単一サブコマンドにしたい
- 歴史的経緯ということで `cmoc review oracle` のままで良いかも
- あとから変えよう

### トークン効率を優先したい

- agent call の並列化とかはトークン効率が落ちるのでやるべきではない
- agent call の細分割もできるだけ避けるべきで、１つの呼び出しで多くの事をまとめてやらせるべき

### 記憶は上手いことリポジトリ内に埋め込みたい

- 普通の memory はうまく行かない傾向がある
- ソースコード・テストが実質的に記憶であるように、所見も記憶の代替としてリポジトリに埋め込むのが良さそう
- 所見リストをファイルとして保持すれば、所見リストに対する操作＝ファイルに対する編集操作なので、うまくいきそう
- 所見リストファイルも git 追跡対象とすれば、git ブランチモデルとの整合性も取れる
- このリスト編集のファイル化のアイディアは他の要素にも持ち込みたい
