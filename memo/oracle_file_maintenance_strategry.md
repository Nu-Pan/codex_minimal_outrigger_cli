
# oracle file メンテナンス戦略

## 前提

- oracle file は人間が直接的に責任を持つ唯一の領域
- cmoc による開発フロー上のボトルネックは、oracle file の手動編集が間に合わないこと
- AI agent によるレバレッジを大きくするには oracle file のメンテナンスの手間の削減は必須

## 人間による最終承認は絶対に必要

- 100% AI は、大局観のないナンセンスな仕様を吐く事がある
- ナンセンスな仕様は別のナンセンスな仕様を呼び込むから、確実に弾きたい
- 「この仕様で良いのか？」とか「この仕様文言でいいのか」みたいな高度な判断は人間で握りたい
- 人間の最終承認を受け入れゲートとして置かざるを得ない

## 簡単な修正は AI に任せたい

- e.g. 誤字脱字の修正、表記揺れの統一
- 人間の最終承認は通す
    - oracle file の編集を AI に任せて確認しないのは怖すぎる
    - cmoc の仕様のシンプルさという意味で例外を作らないほうが良い
    - 「シンプルな修正ならチェックしなくてもズレることはないはず」が成立するかどうかは疑問の余地がある

## 仕様変更のドラフトまでは AI にやらせたい

- oracle file の編集をゼロから人間の手でやるのは認知コストが高い
- AI が書いたドラフトから始められると認知コストが安く済む
- 提案をどうするか（そのまま呑むか、修正を入れて呑むか、参考にだけするか）は人間の裁量

## oracle file が規範に違反していないかのチェックは AI にやらせたい

- 現在の `cmoc review oracles` が該当
- 現在の仕様だとチェックコストが非常に重たいので、簡素化が必要
- 所見リストをステートファイルとして保持して、それを AI に編集させる（追記させる）形式が良いだろう
- 所見リストファイルも git 追跡対象とするなら、`.cmoc` ディレクトリの整理が必要だろう
- 網羅性が重要なので、ファイル単位チェックはやめられない
- ファイル単位チェック後の所見リストの改善を最低限で済ませる必要がありそう

## cmoc のサブコマンド構成は？

- レビューしやすくしたいので、「簡単な修正」と「仕様変更のドラフト」は別のコマンドにしたい
- `cmoc apply fork` みたいな感じになるだろう

## トークン効率を優先したい

- agent call の並列化とかはトークン効率が落ちるのでやるべきではない
- agent call の細分割もできるだけ避けるべきで、１つの呼び出しで多くの事をまとめてやらせるべき
- 入力プロンプトキャッシュも気にする（順序とかの話）

## 未整理

- 所見リストは git 追跡対象にしてインクリメンタルな処理を可能にしたい
- 単純な修正 (typo とか) は自動化して、人間の負担を減らしたい
- apply を実行して仕様変更を反映
- review oracle の裁判モデルに high を使うのは重すぎる
    - １周で良いでしょとか、 FATAL だけで良くない？　とかある
- プロンプトが満たすべき基本要件の定義は関数化したい
- 実際に `cmoc review oracle` を使って oracle の改善を図る
- 所見リストどころか、調査対象ファイルリスト（キュー）もファイルで永続化管理しても良いかも
