# `change_summary.json`

## Summary
- 差分全体を意味ごとの変更カテゴリに分け、人間向けの要約と主要な変更箇所を記録するための構造を定める。
- 変更の網羅一覧ではなく、要約の根拠として有用な主要箇所を添えて、レビューや報告で変更意図を把握しやすくするための出力契約を扱う。

## Read this when
- fork 適用後の差分を、人間が読める変更カテゴリ別サマリーとして出力・検証する処理を実装または確認するとき。
- 変更カテゴリ、変更内容の説明、主要な変更箇所の対応関係について、出力互換性を固定したいとき。
- 差分要約が空でない前提や、詳細な変更一覧ではなく主要箇所を選ぶ方針を確認したいとき。

## Do not read this when
- 個別ファイルのパッチ内容そのもの、diff 生成手順、または git 操作の仕様を確認したいとき。
- fork の作成・適用・分岐制御など、差分要約の出力契約より前後の実行フローを調べたいとき。
- ルーティング文書、テスト、実装などの変更種別ごとの具体的な判定ロジックを探しているとき。

# `change_summary.py`

## Summary
- `cmoc apply fork` の変更差分を、人間向けの要約依頼に変換する組み立て処理。差分本文をそのままプロンプトに埋め込み、返却先の JSON も含めて `AgentCallParameter` を作るため、ここを読むのは fork 変更の要約生成フローを追うとき。

## Read this when
- `cmoc apply fork` の要約生成で、どの入力がプロンプトに入り、どの出力先へ返すかを確認したい。
- 変更要約の生成に使うモデル種別・推論方針・ファイルアクセス条件を知りたい。

## Do not read this when
- 差分の解析ルールそのものを知りたい場合は、要約生成より上流の差分取得側を読む。
- プロンプト本文の共通組み立て規約を知りたい場合は、個別の fork 要約ではなく共通の prompt builder 側を読む。

# `file_finding_enumeration.json`

## Summary
- 実装調査で見つかった問題点を、根拠位置、正本仕様上の要求、観測された実装、問題理由、修正方針として報告するための構造化出力を定義する。
- 仕様と実装の乖離をレビュー結果として人間に渡す場面で使う、所見リストの出力契約を担う。

## Read this when
- 実装レビューや適合性調査の結果として、明確に修正が必要な所見を返す出力形式を確認したいとき。
- 所見に含めるべき根拠情報、仕様要求、観測結果、理由、修正方針の粒度を確認したいとき。
- レビュー結果の生成側または検証側で、所見リストが空でない根拠位置を持つことを前提にしたいとき。

## Do not read this when
- 単に実装対象の仕様そのものを探しているとき。ここには個別機能の要求ではなく、レビュー所見の報告形式だけがある。
- INDEX.md 用エントリーや一般的なルーティング文書の書き方を確認したいとき。
- 所見を JSON 以外の文章、ログ、CLI 表示としてどう見せるかを確認したいとき。

# `file_finding_enumeration.py`

## Summary
- `cmoc apply fork` で、指定ファイルを起点に所見をファイル単位で列挙するためのエージェント呼び出しパラメータ構築を読む対象。プロンプト本文、利用するモデル設定、出力先の決め方を確認したいときに進む。

## Read this when
- `cmoc apply fork` の所見リストアップ用パラメータの内容や、どの前提文脈をエージェントに渡すかを確認したいとき。
- ファイル単位で調査対象を固定し、追加で読むべき oracle file や realization file の条件を知りたいとき。
- 呼び出し設定の強さや、どの出力先に束ねるかを見たいとき。

## Do not read this when
- 所見列挙の実際の判定ロジックや、具体的な修正実装を探したいときは、パラメータ生成先ではなく該当する realization code を読む。
- プロンプト組み立ての共通処理や markdown 化の実装詳細を追いたいときは、このファイルではなく参照先の共通 builder を読む。
- 対象ファイルの内容そのものを評価したいときは、まずそのファイル自身と、必要なら起点候補の周辺本文を読む。

# `finding_application.py`

## Summary
- `cmoc apply fork` の「所見を受けて realization file を修正する」ための agent call パラメータを組み立てる。所見の提示方法、実行時の権限、モデル設定、返却前提を変えるときに読む。
- 同じ apply/fork 配下でも、差分要約や所見列挙ではなく、修正実行側の prompt と呼び出し条件を担当する入口。

## Read this when
- `cmoc apply fork` で、所見リストを受けて修正作業を依頼する prompt や agent call 設定を変えたいとき。
- 所見の渡し方、実装修正を許すファイルアクセス、使う model class や reasoning 効率を確認したいとき。

## Do not read this when
- 変更差分の要約生成を調べたいときは、同じ apply/fork 配下でも変更要約側の定義を読む。
- 対象ファイルの所見列挙ルールや調査対象の選び方を見たいときは、所見列挙側や上位の apply 仕様を読む。
- apply ループ全体の状態遷移、レポート、終了条件を確認したいときは、このファイルではなくサブコマンド仕様を読む。
