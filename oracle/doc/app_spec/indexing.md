
# `cmoc` における `<work-root>` 上の `INDEX.md` の扱い

## 概要

- `<work-root>` 配下の大半のディレクトリ（後述）には `cmoc` によって `INDEX.md` が配置される
- この `INDEX.md` 配置は `cmoc` によって自動的にインデクシングされる（人間は書かなくて良い）

## `INDEX.md` 配置対象ディレクトリ

- `<work-root>` 配下のディレクトリを再帰的に列挙する
- 以下のディレクトリは配置対象から除外する
    - `.` から始まるディレクトリ
    - `.gitignore` の対象
    - `<work-root>/memo`
    - バイナリファイル

## 目次作成対象ファイル・ディレクトリ

- `INDEX.md` と同階層のファイル・ディレクトリを列挙する
- 以下のファイル・ディレクトリは目次作成対象から除外する
    - `.` から始まるファイル・ディレクトリ
    - `.gitignore` の対象
    - `<work-root>/memo`
    - `INDEX.md`
    - バイナリファイル

## `INDEX.md` のフォーマット

- ファイル・ディレクトリ１件ごとに、以下のフォーマットの「目次情報」を用意する
    ```markdown
    # `<ファイル・ディレクトリ名>`

    ## Summary

    <ルーティング先ファイル・ディレクトリの説明>

    ## Read this when

    <どんな作業・質問・変更のときに、このファイル・ディレクトリを読むべきか>

    ## Do not read this when

    <逆に、このファイル・ディレクトリを読まなくて良い・読んではいけないケース>

    ## hash

    <この目次情報を生成するにあたって参照したファイル・ディレクトリとそのハッシュ>
    ```
- `<ルーティング先ファイル・ディレクトリの説明>`
    - 箇条書きとする
    - ファイル・ディレクトリに何が書いてあるか・入っているかを書く
- `<どんな作業・質問・変更のときに、このファイル・ディレクトリを読むべきか>`
    - 箇条書きとする
    - AI がこれを読む判断を下す「条件」を明示する
- `<逆に、このファイル・ディレクトリを読まなくて良い・読んではいけないケース>`
    - 箇条書きとする
    - AI が「関連しそう」程度で過剰に読みに行く傾向を抑制するための「そうじゃないよ」を書く
- `<この目次情報を生成するにあたって参照したファイル・ディレクトリとそのハッシュ>`
    - 箇条書きとする
    - ファイルの場合
        - 単にそのファイルの文字列（中身）から計算したハッシュ値 (SHA-256) を書く
    - ディレクトリの場合
        - 直下の目次作成対象ファイル・ディレクトリをリポジトリ相対パスの辞書順で安定ソートし、各エントリについて `type + '\0' + relative_path + '\0' + content_hash + '\n'` を連結した UTF-8 文字列から SHA-256 を計算する
        
## インデクシングの仕様

### 処理対象

1. 配置対象ディレクトリは前述のルールで機械的に列挙する
2. 列挙されたディレクトリを「深い方から」順番に処理する
    - e.g. `<work-root>/parent/INDEX.md` で `<work-root>/parent/child` の目次情報を作成するために `<work-root>/parent/child/INDEX.md` を閲覧する可能性がある
    - よって、子ディレクトリ側を優先して新鮮な状態にする
3. インデクシングによって発生した git 未コミット差分を自動コミット
    - インデクシングの呼び出し１回ごとに１つのコミットになるようにまとめる事
    - コミット対象は「`INDEX.md` の差分」に限定すること
    - インデクシング開始時点での `INDEX.md` 以外の既存差分はコミット対象から除外すること
    - インデクシング開始時点での `INDEX.md` の既存差分はコミット対象に混ざっても良いものとする
    - 既存差分が存在した場合も処理は続行する

### ディレクトリ１つごとの処理

- `INDEX.md` が存在しなければ新規に作成
- `INDEX.md` 内の目次情報の不備をチェック
    - ファイルシステム上存在するファイル・ディレクトリと対応する目次情報が存在しなければ新規に作成する
    - 存在しないファイル・ディレクトリへの目次情報は削除
    - 目次情報に記録されているハッシュ値と実際のハッシュ値が一致しない場合は目次情報を削除・再生成

### 目次情報の生成方法

- 目次情報の生成対象 1 件ごとに、独立した agent call を行う
- agent call の具体的なパラメータは `build_indexing_index_entry_parameter` を正本とする
- 「ファイル・ディレクトリ名」「ハッシュ値」などの機械的に処理可能な部分は cmoc が責任を持つ

## 目次情報生成の並列実行

- 基本方針
    - 目次情報生成のための `codex exec` 呼び出しは、可能な限り並列に実行する
    - ただし、並列化によって、前述の「深い方から原則」を破ってはいけない
- 並列化規則
    - 同じ `INDEX.md` 内のエントリ同士は並列化して良い
    - 祖先・子孫の関係に無い `INDEX.md` 同士は並列化して良い
        - e.g. `/path/to/A/INDEX.md`, `/path/to/B/INDEX.md` は並列化して良い (親戚なのでセーフ)
        - e.g. `/path/to/A/INDEX.md`, `/path/to/A/B/INDEX.md` は並列化してはいけない (`/path/to/A/B/INDEX.md` を先に処理する)

## インデクシングの実行タイミング

- Codex CLI を実行する前に必ず実行する
- 以下は例外として実行対象から除外する
    - `INDEX.md` を生成するための Codex CLI 呼び出し
    - merge コンフリクト解決用の Codex CLI 呼び出し
- インデクシングは適切に直列化されること
    - e.g. ２つの Codex CLI 呼び出しが起きて、２つのインデクシングが同時起動した場合
    - メンテンナンス処理が排他制御され、直列に実行されること
    - ２つのインデクシングが干渉し合って破綻することはあってはならない

## `INDEX.md` が最新に保たている状態でメンテンナス処理が実行されたら

- 機械的チェックによって、最新状態であることが確認される
- その結果 Codex CLI による目次生成は一切実行されない
