# `cmoc apply` 系サブコマンドについての「やらなかった話」

## 修正点リスト完成後の作業計画立案をやらなかった理由

### 概要

- 「修正点リスト完成 --> 作業計画立案 --> 計画に基づく修正作業」という流れを検討した
- 高度な推論が求められる計画立案作業を切り出すことで、トークン消費を抑えることを目論でいた
- が、結局、この方針は採用しなかった

### 不採用の理由

- 実装作業というのは往々にして「思っていたのと違った」が起きるものであるから、事前の計画と実態が乖離した時に余計な思考にトークンが割かれてしまうリスクが有る
- 次のループの修正点リストアップで回収される事を考えると、修正作業の精度を上げる必要性が無い
- 抜け漏れを気にするのであれば修正点リストアップをもっとリッチにすれば良い


## 所見リストの並列リストアップをやらなかった理由

### 概要

- 旧 `cmoc apply fork` では、調査対象ファイルを先に列挙し、各ファイルを起点にした所見リストアップを並列実行する方針を検討していた
- ファイルごとの所見リストを連結したあと、agent call で所見リスト全体を改善する
- 改善後の所見リストに含まれる所見 1 件ごとに、独立した agent call で修正作業を行う
- さらに、候補ファイルごとのダーティフラグで、次に調査すべきファイルを管理する想定だった

### 不採用の理由

- 所見調査時と適用時とでは realization file の状態がズレていている可能性がある
- 複数ファイル間での所見の重複発見が起きて無駄が発生し、その無駄をリダクションするために余計な agent call が必要になって、トークン消費がかさむ
- 並列リストアップは、ファイル間の依存関係を agent call の外側で先に固定するので、実際の修正作業で見つかる関連ファイルや影響範囲を反映しづらい
- 所見リスト全体の改善ループは、重複や False-Positive の除去には使えるが、所見を正規化するためだけの独立した orchestration と prompt を増やす
- 所見 1 件ごとの修正は、複数所見が同じ実装箇所へ関係する場合に、agent call 間の文脈分断とコミット粒度の細分化を招く
- ダーティフラグ方式は、apply 中に変化する調査対象を表現するには重く、現在の「調査待ちファイルリストへ追加して処理する」方式の方が単純である
- そのため、旧仕様の「列挙を先に広げ、改善してから所見単位で直す」流れは採用せず、ファイル単位で所見調査と反映作業を進める方式に寄せた
