# `apply_behavior.md`

## Summary

- `cmoc apply` 系サブコマンドで、修正点リスト完成後に作業計画を立てる案を採用しなかった理由を整理した文書です。
- 事前計画は実装時のズレに弱く、かえって余計な思考コストを生むため見送ったという判断を述べています。
- 抜け漏れ対策は計画を増やすより、修正点リストをよりリッチにする方向で扱う立場を示しています.

## Read this when

- 修正点リストの完成後に、さらに作業計画を立てる設計を採るべきか判断したいとき。
- 計画を事前に切り出すことでトークン消費を抑えられるか、実装との乖離リスクが増えないかを検討したいとき。
- 抜け漏れ対策として計画を増やすべきか、修正点リスト側を厚くするべきかを比較したいとき。

## Do not read this when

- `cmoc apply` 系の実装手順やコマンド仕様そのものを確認したいときは、該当する正本仕様を直接読むべきです。
- 作業計画レビューや `eval-oracle` の採否を、設計方針全体として検討したいだけのときは、この文書ではなく関連する別文書を優先すべきです。
- `INDEX.md` の生成ルールや `oracle` 全体のルーティング方針だけを確認したいときは、この文書を読む必要はありません.

# `working_plan_review.md`

## Summary

- `tgbt plan` や `/plan` のような計画レビュー方式を採らなかった理由を説明する文書です。
- 人間が最終成果物に関心を持ちやすいこと、AI への丸投げが成立しにくい現実、そして人間と AI を同列に扱う設計の不自然さを整理しています。
- 結論として、人間が `oracle` を編集し、AI が実装を追従させるスタイルを採用した背景をまとめています。

## Read this when

- 作業計画レビューを採用しなかった理由を確認したいとき。
- 人間が `oracle` を編集し、AI が実装を追従する方針の背景を知りたいとき。
- 人間と AI を同じ土俵で共同作業させる設計の問題点を把握したいとき。

## Do not read this when

- `tgbt plan` や `/plan` の採否を判断したいとき。
- 人間と AI の役割分担をどう設計するかを一般論として議論したいとき。
- `cmoc eval-oracle` の採用理由ではなく、実装手順やコマンド仕様だけを確認したいとき。
