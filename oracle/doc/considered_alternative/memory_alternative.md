# AI-generated kaizen を自動的に次回実行へ反映しない理由

cmoc では、AI に実行結果を振り返らせ、「次回以降はこう改善すべき」という kaizen を生成させ、それを自動的に後続の Codex CLI 呼び出しへ注入する仕組みは採用しない。

これは、振り返りや改善案そのものを否定する判断ではない。問題は、AI が生成した改善案を、検証・整理・人間による採用判断なしに、次回以降の実行コンテキストへ継続的に混ぜる点にある。

この仕組みは一見すると、AI が経験から学習して賢くなるように見える。しかし実際には、仕様、ログ、一時的な失敗分析、思い込みが混ざった、曖昧な準仕様レイヤーを作ることになりやすい。

cmoc は、Codex CLI を外側から制御するための最小限の補助ツールである。そのため、動作の根拠はできるだけ明示的な仕様、明示的な入力、明示的なログ、明示的な成果物に置くべきである。

AI-generated kaizen を自動注入すると、どの指示が現在の挙動に影響しているのか追跡しづらくなる。ある実行でたまたま有効だった対処が、別の実行では逆効果になる可能性もある。

また、AI による失敗原因の分析は常に正しいとは限らない。誤った原因分析が kaizen として保存されると、以後の実行で同じ誤解が繰り返し参照される。これは改善ではなく、誤診断の永続化である。

特に cmoc では、正本仕様断片として oracle を持つ。AI-generated kaizen が自動的に読まれるようになると、oracle とは別に、暗黙の仕様が存在する状態になる。これは仕様の単一性を壊す。

さらに、kaizen は蓄積するほど古くなる。古い制約、古い実装、古い失敗、古い回避策が残り続けると、現在の状況に合わない指示が混入する。それを自動的に pruning することもまた難しい。

Codex CLI 本体にも memory 系の仕組みは存在するが、スコープ、更新タイミング、要約、再生成、利用量消費などの点で、運用上の難しさがある。cmoc が同種の仕組みを独自に持つと、同じ問題を別レイヤーでも抱えることになる。

cmoc が目指すべきなのは、AI に暗黙記憶を持たせることではない。むしろ、AI が必要な情報へ明示的に到達できるようにすることである。そのためには、INDEX、oracle、ログ、実行成果物を整備する方がよい。
